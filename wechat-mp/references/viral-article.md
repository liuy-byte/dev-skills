# 单篇爆款文章（通用写作规范事实源）

将任意技术文章或参考内容（URL 或本地 HTML）改写成微信公众号风格的爆款单篇文章。本文档同时是整个 skill 的**通用写作规范事实源**——标题/语言/禁忌/图床 URL/发布闭环均定义于此，连载写作沿用。

> 写连载系列见 `series-writer.md`；规划、生成或规整正文配图见 `body-images.md`；生成封面见 `cover.md`；传封面图拿 `media_id` 见 `media-upload.md`。

## 工作流程

### 第一步：获取内容

**优先使用 curl 下载 HTML 到本地**，不要依赖 WebFetch（可能被安全策略拦截）：

```bash
curl -L "<URL>" -o "/tmp/article-<timestamp>.html"
```

如果用户提供的是本地 HTML 文件路径，直接读取。

### 第二步：自适应提取正文和图片

不同网站的 HTML 结构不同，使用以下策略逐级尝试：

**提取正文（按优先级尝试）：**

1. 找 `<article>` 标签
2. 找 `class` 包含 `article`、`content`、`post`、`entry`、`markdown-body` 的 `<div>`
3. 找 `<main>` 标签
4. 找最大文本容量的 `<div>`（通过比较子文本长度）

```python
# 伪代码思路
soup = BeautifulSoup(html, 'html.parser')
article = soup.find('article') or \
          soup.find('div', class_=re.compile(r'article|content|post|entry|markdown')) or \
          soup.find('main') or \
          max(soup.find_all('div'), key=lambda d: len(d.get_text()))
```

**提取图片：**

1. 先从 `<meta property="og:image">` 获取文章封面图
2. 再提取正文区域所有 `<img>` 标签的 `src` 或 `data-src`
3. 过滤掉图标、Tracking pixel 等小图（宽或高 < 200px）
4. 下载时优先获取最高清版本（去掉 URL 中的 `resize` 参数）

```bash
# 去掉阿里云/七牛等图床的 resize 参数示例
url="${url//\?x-oss-process=image\/resize,w_*/}"
```

### 第三步：分析并规划文章结构

读懂原文后，确定：

1. **文章核心主题**是什么
2. **目标读者**是谁
3. **最有传播价值的点**在哪里（效率提升？工具推荐？避坑指南？）
4. **需要几张配图**，为每张图确定作用和插入位置；来源图片不足或需要原创图时读取 `body-images.md`

### 第四步：准备正文配图

在项目的 `images/` 子模块下为文章建一个专用文件夹：

```bash
mkdir -p "<项目路径>/images/<文章专属文件夹>/"
```

优先下载最有价值的 3-5 张来源配图：

```bash
curl -L "<图片URL>" -o "<项目路径>/images/<文章专属文件夹>/<图片名>.png"
```

来源配图不足、图片与改写后的正文不匹配，或用户要求原创、修复、统一尺寸时，按 `body-images.md` 规划、生成、编辑并校验。技术教程优先真实截图和确定性图形，不用无关的抽象 AI 插图填充数量。

### 第五步：按公众号风格重写

**这是核心步骤。严格按照以下规范执行：**

#### 标题规范
- 以**数字**量化价值：「16个」「5步」「3倍」
- 以**动作词**制造张力：「干翻」「告别」「别再...」
- 包含**具体收益**：「效率翻倍」「少踩5个坑」
- 格式：`[产品/工具]｜[核心价值/卖点]` 或直接点明效果
- 当前日期格式：`YYYYMMDD`

**好标题例子：**
- `Claude Code｜别再乱用了，这16个命令用顺了效率差很多`
- `Ghostty + Claude Code：终端开发最佳搭子`

**坏标题例子：**
- `Claude Code 使用指南`
- `关于 Claude Code 的一些命令介绍`

#### 开头规范（引言块）

默认情况下，第一段使用 **blockquote（引用块）**，2-3 句话，不超过 3 行。

**但如果仓库内的 CLAUDE.md、记忆或用户明确要求不用引言块，必须优先服从用户/项目规范。**

**核心逻辑：描述读者痛点**，让读者觉得"说的就是我"。

**模板：**
```
> [描述一个真实痛点场景]，
> [点名核心问题或误区]。
```

**例子：**
```
> 很多人用 Claude Code，用着用着就觉得：也就这样。
> 真不是工具不行。大概率是你把它当成了终端版聊天框。
```

**禁止：**
- ❌ 模板化的"本文价值"列表
- ❌ "担心...？想...？"引导句式
- ❌ "总的来说"、"综上所述"

#### 文章结构

汉字编号：一、二、三...

主要章节之间用 `---` 分隔。

**每个功能/概念独立成节，用 ✅ 列表突出核心能力。**

**实战案例必须包含：具体场景 + 执行过程 + 量化效果。**

```markdown
> **实战案例**：[具体场景]，[工具/方案]完成了[具体操作]，[量化结果]。
```

**例子：**
```markdown
> **实战案例**：给登录模块加 OAuth 支持时，先在 Plan Mode 里梳理调用链、拆改动点，再进入执行模式，返工会少很多。
```

#### 效率数据规范

- **必须具体**：「提升 3.5 倍」而非「大幅提升」
- **必须有实战支撑**，不能随意捏造
- **保留原文量化数据**，原文没有的不自己编

#### 图片引用规范

正文中**必须使用图床 raw URL**，不使用本地相对路径。图床前缀从 `~/.config/wechat-mp/config.json` 的 `image_bed.prefix` 读取（与素材上传共用同一配置文件）；缺失时先询问用户其图床仓库，再写入：

```json
{ "image_bed": { "prefix": "https://raw.githubusercontent.com/<user>/<images-repo>/main" } }
```

拼接规则：`<prefix>/<文章专属文件夹>/<图片名>.png`

⚠️ **URL 中不含 `images/` 层级**。图片虽然存放在本地 `images/` 子模块目录下，但 raw URL 直接是 `<prefix>/<文章专属文件夹>/`。

- ✅ 正确：`<prefix>/navicat-win-install/start-install.png`
- ❌ 错误：`<prefix>/images/navicat-win-install/start-install.png`（多了 `images/`，会 404）

在正文中插入图片：

```markdown
![图片说明](<图床URL>)
```

#### 语言风格

- **短句为主**，节奏感强，避免长难句
- **口语化但专业**：技术术语保留英文，解释用中文
- 使用 emoji 辅助排版（✅ ❌ ⚡ 💡 🔥），不过度堆砌
- 步骤说明用简洁陈述句，不逐条拆解
- **多用自己的观察和判断**，少用"整理""总结""教程"
- 避免"真正拉开差距""隐藏王牌""最强的地方"等模板化表达
- 多用"我自己""很多人问题出在""越来越喜欢"等第一人称表达

#### 格式规范

- 主要章节之间用 `---` 分隔
- 对比类内容优先使用表格
- 代码块内配简短中文注释
- 图片统一放 `images/` 子模块目录，正文使用图床 URL

#### 禁忌清单（严格避免）

- ❌ 模板化的"本文价值"列表（✅ 真实环境实操 / ✅ 完整命令行输出...）
- ❌ "担心...？想...？"引导句式
- ❌ 写了表格又在正文重复说明
- ❌ 章节标题加"解读"、"说明"等冗余词
- ❌ "这是**唯一需要提前准备**的内容"等强调废话
- ❌ 「如有疑问欢迎留言」等无效客套
- ❌ 「总的来说」「综上所述」等套话过渡
- ❌ 效率数据无实战支撑随意捏造
- ❌ AI 味表达：过整齐、过标准答案感、过"教学讲义腔"
- ❌ 结尾加 GitHub 仓库链接、版权声明 blockquote
- ❌ 为了"简洁"压缩实操教程内容——实操教程每一步都有必要，不应以压缩字数为目标

#### 文章末尾模板

```markdown
---

## 相关链接

微信内长按复制链接到浏览器打开。

**[分类名]**
​```
链接名：https://example.com/
​```
```

### 第六步：去除 AI 味（如果用户要求）

如果用户明确说"去除AI味"、"更像人写的"，在完成上述步骤后，额外做以下调整：

**去掉：**
- "真正拉开差距""隐藏王牌""真正顺手的"等模板化表达
- 过整齐的列表和总结
- "教学讲义感"和"知识整理稿感"

**加强：**
- 第一人称观察和判断（「我发现...」「很多人问题出在...」）
- 具体的、个人化的使用场景描述
- 更口语、更直接的短句
- "这个命令看起来像准备动作，其实最省时间"这种只有用过才知道的判断

**目标：读起来像一个有经验的开发者在朋友圈/社群分享经验，不是"标准答案式"教程。**

### 第七步：保存文章文件

文件命名：`YYYYMMDD-标题.md`（年月日为前缀，标题简洁）

保存在 `articles/` 子模块根目录（不是 `images/` 或 `docs/`）。

### 第八步：发布前自检（必做）

文章保存后，**逐条核对**，不通过先改再发：

- [ ] 图床 URL 格式正确：与配置的 `image_bed.prefix` 一致、含 `<文章专属文件夹>/`、**不含 `images/`**
- [ ] 所有量化数据都有实战/原文支撑，没有凭空捏造
- [ ] 没有禁忌清单违例（模板化"本文价值"、"担心…想…"句式、套话过渡、无效客套、结尾 GitHub/版权块）
- [ ] 没有 AI 味（过整齐列表、标准答案腔、教学讲义感）
- [ ] 字数较 AI 初稿精简（实操教程除外，教程不为省字压缩步骤）
- [ ] 文件命名 `YYYYMMDD-标题.md`，已保存在 `articles/` 根目录
- [ ] 正文配图已下载或生成到 `images/<文章专属文件夹>/`，数量、相关性、尺寸和文字准确性通过 `body-images.md` 检查

**重点文章**（要重点推的）：可以让我再独立通读一遍，对照上面逐条挑问题，而不只是机械打勾。

### 第九步：发布闭环（准备命令，用户确认后执行）

**重要：子模块和父仓库必须一起提交，否则父仓库引用不更新。以下命令交给用户确认后执行，不自动 push。**

**① 推文章（articles 子模块）：**
```bash
cd articles
git add *.md && git commit -m "update: <文章标题>" && git push
cd ..
```

**② 推配图（images 子模块）：**
```bash
cd images
git add . && git commit -m "chore: add <图片说明>" && git push
cd ..
```

**③ 父仓库 bump 子模块引用：**
```bash
git add articles images && git commit -m "chore: update articles & images - <文章标题>" && git push
```

**④ 封面图（发到公众号时需要）：**

用素材上传脚本传封面拿 `media_id`（详见 `media-upload.md`）：
```bash
cd ~/.claude/skills/wechat-mp/scripts/media
node upload_media.mjs --image <封面图绝对路径>
```

微信头条封面使用 2.35:1；本 Skill 默认生成 2350×1000 高清母版，文件 ≤2MB，并上传为永久素材。发布前检查横图和 1:1 分享缩略图裁切。

## 输出

1. **文章文件**：`articles/YYYYMMDD-标题.md`
2. **配图文件夹**：`images/<文章专属文件夹>/`
3. **图床 URL**：正文中使用的图片链接
4. **发布命令清单**：子→母仓库提交命令 + 封面上传命令（用户确认后执行）
