# Git Commit

创建与用户意图一致、可审查且可回滚的 Git 提交。优先遵循仓库自己的提交规范；没有明确规范时使用 Conventional Commits。

## 工作流

### 1. 读取约束与现状

1. 确认当前目录属于 Git 仓库，并读取适用的 `AGENTS.md`、`CONTRIBUTING.md` 或其他提交约定。
2. 运行 `git status --short --branch`，识别当前分支、暂存、未暂存、未跟踪文件及冲突状态。
3. 查看 `git diff --stat`、`git diff`、`git diff --cached` 和近期 `git log --oneline`，理解改动目的与仓库现有提交风格。
4. 将实际改动与用户本次提交意图对齐。不要假定工作区中的所有修改都属于本次任务。

### 2. 确定提交边界

- 一次提交只表达一个逻辑变更。若改动包含多个独立主题，给出推荐的拆分方案并请用户选择后再提交。
- 只暂存本次提交需要的文件或代码块，保留用户已有的无关改动。不要为了整理工作区而丢弃、覆盖或还原文件。
- 暂存未跟踪文件前先确认其用途；不要提交密钥、令牌、私钥、`.env`、临时文件、构建产物或异常的大型二进制文件。
- 若用户明确要求“提交全部改动”，仍需先检查全部改动是否相关、完整且安全。

遇到以下情况时停止提交，说明证据并给出建议：

- 存在未解决的合并、rebase、cherry-pick 或冲突标记；
- 改动明显未完成，或包含与用户意图不一致的内容；
- 疑似包含秘密信息、个人数据或不应入库的生成文件；
- 无法可靠区分应提交与应保留的改动。

### 3. 提交前验证

1. 运行仓库明确要求且与改动相关的检查。不要擅自执行耗时、产生外部影响或需要秘密凭据的任务。
2. 至少运行 `git diff --check`；暂存后再运行 `git diff --cached --check`。
3. 验证失败时不要绕过检查或提交明显有问题的内容。报告失败原因，并建议修复后提交。

### 4. 生成提交信息

仓库规则和用户明确要求优先。没有其他约定时使用：

```text
<type>(<scope>): <subject>

<body，可选>

<footer，可选>
```

选择准确的 `type`：

| type | 用途 |
| --- | --- |
| `feat` | 新增用户可见功能 |
| `fix` | 修复缺陷 |
| `docs` | 仅修改文档 |
| `style` | 不影响行为的格式调整 |
| `refactor` | 不新增功能、不修复缺陷的代码重构 |
| `perf` | 性能优化 |
| `test` | 新增或修改测试 |
| `build` | 构建系统或依赖变更 |
| `ci` | CI 配置或脚本变更 |
| `chore` | 其他维护性改动 |
| `revert` | 回滚已有提交 |

- `scope` 仅在能准确表达受影响模块时使用；跨多个模块时可以省略。
- `subject` 使用祈使语气，明确说明“做了什么”，保持简洁，不以句号结尾。语言跟随用户要求或仓库近期风格；无法判断时默认使用简体中文。
- `body` 解释变更原因、关键取舍及行为差异，不要只罗列文件名。
- 使用 `type(scope)!:` 或 `BREAKING CHANGE:` 标记破坏性变更。
- 仅在有明确依据时添加 `Closes #123`、`Refs #123` 等页脚。
- 不添加 `Co-authored-by`、AI 生成声明或其他用户未要求的署名。

### 5. 创建并核验提交

1. 使用精确路径或安全的分块方式暂存目标改动，避免无判断地执行 `git add .`。
2. 再次检查 `git diff --cached --stat`、`git diff --cached --name-status` 和完整暂存差异，确认没有漏项或夹带内容。
3. 以非交互方式执行 `git commit`，保留 Git hooks 的正常检查。
4. 提交后运行 `git show --stat --oneline --decorate HEAD` 和 `git status --short --branch`。
5. 向用户报告提交哈希、提交标题、包含范围、验证结果及剩余未提交改动。

若没有可提交的改动，直接说明，不创建空提交。

## 操作边界

- 默认只创建 commit，不执行 `git push`。仅在用户明确要求推送时继续。
- 不自动执行 `--amend`、rebase、reset、强制推送或跳过 hooks；这些操作必须由用户明确要求。
- 不修改业务代码来“凑成”一次提交；若提交前检查发现问题，先报告并等待用户决定是否修复。
- 不因提交任务清理用户工作区，也不更改与本次提交无关的分支或远端配置。
