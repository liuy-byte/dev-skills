# 批量派单模式(方式 A)完整操作规程

> **何时读本文件**:一次要修 ≥2 个云效工单、按 SKILL.md「作业模式」一节选定批量派单模式时。单工单亲跑不需要读。

## 操作规程(7 条)

1. **主会话只做四件事**:①按路由判定每单归属 → ②写清派单 prompt → ③串接多角色 / 汇总结论 → ④把**需要人定案的决策点**收上来给用户。主会话**不亲自**读工单正文诊断、不亲自连库、不亲自改码——这些全在 subagent 内完成,避免污染主上下文。
2. **按项目 CLAUDE.md / 路由表选 `subagent_type`**(路径即归属;若当前工作区定义了角色路由表,以它为准)。拿不准归属先让 subagent 只做「阶段 1 拉取 + 阶段 2 分诊」回传,再据结论决定派谁修。
3. **派单 prompt 必须自带全部上下文**(subagent 看不到主线):写清 **①工单号 + organizationId + projectId ②要它走本 SOP 哪几个阶段(通常 1→8,部署留主会话)③改哪些文件 / 范围边界 / 验收点 ④相关红线**(尤其「逻辑/数据类必须连库验真、只读优先 / 写验证事务内 ROLLBACK、严禁生产库写」「越界改动停下报告、只改本工单范围」)⑤要求**回传结构化结论**(根因一句话 / 改了什么 / 连库验证实际结果 / 是否命中早停岔路 / 待用户定案的点 / commit 情况)。
4. **⚠️ 多 subagent 改同一仓库 → 每个必用独立 worktree**(血泪教训,见 SKILL.md 铁律与 worktree-and-sorting.md)。用 `Agent`/`Task` 工具时直接传 `isolation: "worktree"` 让运行时给每个 agent 开独立 worktree;派单 prompt 里写死「你在 worktree `<path>` 内工作,不切分支、不 stash 主仓库」。不同仓库天然隔离、无需 worktree;**同一仓库**多 agent 才需要。
5. **互不依赖的工单同一轮并发派**(同一条消息里发多个 `Task`),不要串行等待;有依赖的(如同一破坏性变更波及多端)按依赖串。
6. **决策点必须回主会话,不甩给 subagent 全权拍板**:①**口径冲突 / 产品需求变更 / 大重构**(阶段 4-B/C、阶段 2 岔路②)——subagent 只负责把冲突点和候选口径**查清回传**,是否定案、怎么定,回到主会话由用户/产品拍;②**对外动作**(转工单状态、留言、git push、部署)——subagent 可**准备好**(写好留言草稿、备好 commit),但**实际执行前先回主会话让用户确认**(除非用户已明确「照 SOP 自动处理」)。
7. **主会话汇总 + 收口**:每个 subagent 回来后,主会话把结论**简要**汇总给用户(每单一两行:修没修好 / 命中什么岔路 / 待定案什么);逐单确认后再由主会话统一做对外动作(转态、留言、push、部署)。**批量修完一轮后,提醒用户 `/clear`**(见 SKILL.md 铁律「一工单一会话」)。

## 为什么决策点不下放

subagent 之间无共享上下文、不能互相交接,更没有跨工单的产品口径全局视角;口径类 / 破坏性变更一旦让 subagent 自行拍板,极易各自为政、与已上线逻辑打架(实践中大量"口径反复""跨端契约"教训)。subagent 擅长的是**在隔离上下文里把一个工单的诊断+修复+连库验证跑扎实**,不是替产品做决定。
