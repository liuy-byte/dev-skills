# 部署与大重构(参考)

> 主流程见 SKILL.md;本文补两类超出「改一个 bug 就提交」的场景:**① 云效流水线部署实操**、**② 工单其实是跨端大重构 / PRD 的分阶段落地**。均为实战沉淀,工具名以云效 MCP 实际返回为准。

---

## 一、云效流水线部署实操

### 1. 别默认「push 会自动部署」——先判触发方式

很多流水线是**手动触发型**,push 后不会自动跑。判别:

```
list_pipelines(organizationId, pipelineName="<仓库名关键词>")   # 拿 pipelineId
get_latest_pipeline_run(organizationId, pipelineId)             # 看最新一次运行
```

看返回的:
- `triggerMode`:`1`=手动(页面点)、`4`=API 触发、`3`=Code commit(push 触发)、`2`=定时、`6`=Webhook。**只有 `3` 才是 push 自动跑。**
- 最新运行部署的 commit(`CI_COMMIT_SHA` / `sources[].commit`):**若不是你刚推的那个,或没有一条 push 触发的新运行 → 代码推了但没部署,环境跑的是旧包。**

> 反例(本次真实踩坑):以为 push 就自动部署了,结果 `yly-backend`/`yly-nurse-frontend`/`yly-frontend` 三条流水线全是 `triggerMode:1` 手动型,push 后最新运行还是几小时前的旧 commit——环境根本没更新。必须手动触发。

### 2. 手动触发部署

用户确认要部署后:

```
create_pipeline_run(organizationId, pipelineId, branch="dev", comment="部署<工单号><改了啥> to dev")
# 返回 pipelineRunId(一个数字,如 105)
```

- 用简化参数 `branch`(最常见),别自己拼 `params` JSON。
- `comment` 写清部署内容,便于事后追溯。

### 3. 轮询到部署真完成

```
get_pipeline_run(organizationId, pipelineId, pipelineRunId)
```

- 顶层 `status` 走 `RUNNING` → `SUCCESS`/`FAIL`。
- **构建 stage 和部署 stage 都要 SUCCESS**;部署 stage 里 `deployOrder.status:Success` 才算主机部署+重启(`systemctl restart` / nginx 覆盖)真完成。只有构建 SUCCESS、部署还 INIT/RUNNING 不算好。
- 全量 `mvn clean install` 首次较慢(3-5 分钟),node 构建 1-2 分钟。**别用前台 `sleep` 干等**(会被拦),用回查(ScheduleWakeup)或后台轮询;缓存窗口内(<5min)可稍密,超时的用长回查。
- `FAIL`:结果里每个 job 的 `actions` 有 `LogPipelineJobRun` 的 `downloadUrl`——取日志报告用户,别只说"失败了"。

### 4. 多端部署:后端先行

跨端改动(前端详情页/家属端依赖后端新接口):
1. **先部署后端** → `get_pipeline_run` 确认 SUCCESS(接口就绪)。
2. **再部署前端**(各前端仓库之间无依赖,可并行)。

否则前端先上、后端没好 → 短暂拿旧数据/报错窗口。这与「阶段 2.5-C 大重构:后端接口先行」一致。

### 5. 部署是 reopen 工单的头号隐藏根因

上轮"已修复"评论在、代码在远程 dev,但手动型流水线没人点部署 → 环境旧包 → 测试复现如初 → reopen。**复查工单先按此排查**:先确认上轮 commit 在不在远程 dev(`git fetch` 后 `git merge-base --is-ancestor <sha> origin/dev`)、再确认流水线最新运行有没有部署这个 commit。往往"没改错,只是没部署"。

## 二、工单是跨端大重构 / PRD 的分阶段落地

### 识别信号

- 工单正文是**一整份需求文档**(背景/目标/范围/多个"待产品确认项"),不是"复现步骤/实际/预期"三栏。
- 诊断后发现要**跨多端 + 后端、删页 / 改数据源 / 改已上线口径 / 破坏性变更**。

这类**不能当 bug 一把梭**,否则改坏面极大且边界没定。按下面分阶段。

### 阶段化范式

**第 0 步 · 出方案(委派 product-architect 视角)**,产出文档含:
- **目标与范围**(本次做什么 / 明确不做什么)。
- **边界决策清单**:把每个"待产品确认项"做成「问题 + 候选方案 A/B/C + 推荐及理由 + 影响」,让产品**逐条拍板**。这是文档核心。
- **跨端接口契约**:新增/改动接口的出入参字段表;守 `AjaxResult{code,msg,data}` 外层不变、只扩 `data`、向后兼容;口径/枚举一律后端返 `code+text` 成对,前端只读兜底,不各端硬编码(防漂移)。
- **三端 + 后端改动清单**(标 `[新增]/[改造]/[下线]` + 依赖关系)。
- **实施顺序**:后端先行 → 前端各端并行 → 下线/删除最后。
- **风险与回归**:破坏性点、必须回归的保留功能清单、数据迁移风险。

留言把方案链接/要点给产品,**决策点定案后再动手**。

**第 1 步 · 后端打地基**(前端阻塞依赖):新接口/出参、口径切换、枚举文案映射、数据盘点 SQL。同模块多改动怕文件冲突时,先做公共依赖(如枚举常量类)、再并行其余。

**第 2 步 · 前端各端并行对接**:不同仓库/不同接口的端(如护理端 vs 家属端)完全并行;同端内有依赖的(详情页依赖列表)串行。

**第 3 步 · 老逻辑下线(最后做,走 Gate)**:
- **下线 Gate**(全绿才做):①读路径全切完(grep 全仓无对老接口的活跃调用)②相关数据清零(如老模型在册数据盘点为 0)③历史处置已定(老表只读归档)。
- **分两步**:先摘前端入口(移除页面注册 + 清残留跳转,页面文件可留着不物理删)观察一迭代 → 再禁后端接口(返回 410「已下线」+ `@Deprecated`,**保留方法体注释掉备回滚**,不物理删)。
- **甄别别盲删**:下线前 grep 老接口的调用方与写入来源。常有**新主入口仍活跃调用某个"老"接口**的情况(如入口合一后的新页底层还调老的某接口)——这种**暂缓下线**,注释说明依赖,待其切新链后再下。老表停写= grep 确认写入来源都来自已下线接口即自然停写,不删表、只读归档。

### 只有 dev 单环境时「跑顺流程」

若项目只有 dev 一个环境、目标是把整条重构流程在 dev 跑通(而非纠结生产级运营补录):
- 下线 Gate 卡在"还有 N 条老数据没迁"时,若那些都是**测试脏数据**(孤儿数据 / 测试老人的测试药),可**直接软删让 Gate 清零**(`UPDATE ... SET del_flag=2`,先 SELECT 留底、确认是脏数据、连的是 dev 库),而不是走真实的 med-inbound 逐条补录。
- 改数据前**先跟用户确认**(即便 dev、即便测试数据):列出要清哪几条、为何是脏数据、软删可回滚。
- 盘点 SQL 复跑返回 0 行 = Gate 清零,再走阶段 3 下线。

### 连库操作 / 验证的安全红线(贯穿盘点·下线·软删·验真)

> 大重构里大量动作要**连真实数据库**:数据盘点 SQL、下线 Gate 的"数据清零"核对、清测试脏数据的软删、以及每个阶段改完后**连库确认结果符合预期**(这与主 SOP「阶段 6.2 连库验证」是同一条强制门禁——逻辑/数据类改动**未连库验真前不得声称完成 / 转「已修复」**,构建通过只代表能编译)。连库时守住红线,别让盘点/验证本身搞坏数据:
>
> - **先确认连的是哪个库**:动手前查清是 **dev / 测试库**还是**生产库**(别只信 profile——可插独特账号实测)。大重构常只在 dev 跑,更要坐实。
> - **只读查询优先**:盘点、核对、验真能用 `SELECT` 看清的,绝不写库。下线 Gate 的"读路径全切完 / 数据清零"核对全是只读盘点 SQL。
> - **必须写验证 / 改数据时,用事务内复现 + `ROLLBACK` 不落库**;确需真改(如软删脏数据让 Gate 清零)则**先 SELECT 留底、确认是脏数据、确认连的是 dev 库,再改**,且改前跟用户确认(见上「只有 dev 单环境」一节)。
> - **严禁在生产库做写验证 / 造数**:生产库一律只读。
> - **临时造的验证数据 / 改的测试账号状态,验完清理**,别留脏数据污染后续联调与盘点。

### 工单流转时机

大重构三阶段实现+部署 dev 后,可流转「已修复」+ 留言(说明各阶段 commit、部署运行号、刻意保留的遗留、回归清单路径);但整体真正完成还需真实账号联调 + **逻辑/数据类连库验真(见上「连库操作/验证的安全红线」)**,按「测试服实测后再关单」的一贯偏好,也可先不流转、等联调 + 连库验证通过再关——由用户定。
