# 连库验证与踩坑案例(参考)

> 本文档是 yunxiao-bug-fix SOP 里「逻辑/数据/口径类 bug 必须连库验真」这条强制门禁的详细展开与项目专属案例细节,主流程见 SKILL.md 铁律段与阶段 6。以下各节均从 SKILL.md 原文剪切而来(一字未删):铁律段的连库验真条与安全红线条、阶段 3.3 的连库坐实展开、阶段 6.2/6.3 的分步骤与项目案例(`t_duty_assignment` / `caregiver_id` / `t_med_batch` / NEBZ297/314/317/425 等)。

## 一、铁律层:逻辑/数据/口径类 bug 必须连库验真(强制门禁·完整展开)

- **⚠️ 逻辑/数据/口径类 bug 必须连库验真,禁止凭感觉修复(强制门禁)**:凡是涉及**业务口径 / 算法计算 / 归属判定 / 校验规则 / 状态流转 / 数据显示不符**这类"逻辑问题"(区别于纯前端视觉/交互),**三个环节都要连真实数据库确认,只读代码推演不算数**——
  - **① 诊断时(阶段 3)**:连库查**真实数据**坐实根因,别只靠读代码脑补。工单说"值不对 / 派给了错的人 / 少了记录 / 显示成 X",就去库里把那条真实记录、关联表、字段实际值查出来,让根因**被真实数据解释**(如 NEBZ297/314/317/425 等,都是靠 DB 实测才坐实病根 / 排除误报)。
  - **② 修复后(阶段 6)**:连库确认**改完后的结果符合预期**——重新走一遍触发条件,看库里落的数据 / 计算出的值 / 归属判定是否真的对了。写路径改动尤其要在库里验证副作用(是否漏更新、是否多写、是否影响到别的记录)。
  - **③ 门禁**:**未连库验真前,不得把工单转「已修复」**、不得在留言里声称已修复。构建通过只代表"能编译",不代表"逻辑对了"。
  - **豁免**:纯前端视觉 / 交互 / 样式类(颜色、布局、动画、图标),无后端数据参与的,可豁免连库,按阶段 6 走查即可。拿不准是否属逻辑类时,从严——按需连库。

## 二、连库验证的安全红线(防止验证过程本身搞坏数据·完整展开)

- **⚠️ 连库验证的安全红线(防止验证过程本身搞坏数据)**:
  - **先确认连的是哪个库**:动手前务必查清当前要连的是 **dev / 测试库**还是**生产库**(参见记忆里"先实测后端真实连哪个库"教训——别只信 profile,可插独特账号实测)。
  - **只读查询优先**:能用 `SELECT` 看清楚的,绝不写库。
  - **必须写验证时,用事务内复现 + `ROLLBACK`**:把写操作包在事务里,验完立即回滚,**不真正落库**(参见记忆里多起"事务内复现 + ROLLBACK"做法,如护理端告警可见性、NEBZ425 CAS 抢单验证)。
  - **严禁在生产库做写验证**:生产库一律只读;确需写验证只在 dev / 测试库进行。
  - **测试账号 / 造的数据验证后清理**:为验证插的临时数据、改的测试账号状态,验完清理干净,别留脏数据污染后续测试。

## 三、阶段 3(根因定位)连库坐实的详细步骤与项目案例

> 下为 SKILL.md 阶段 3 第 3 条原文(逻辑/数据/口径类连库坐实根因)剪切:

3. **⚠️ 逻辑/数据/口径类:连库查真实数据坐实根因,别只靠读代码推演**(强制,见「贯穿全程的铁律」)。工单说"值不对 / 派给了错的人 / 少了记录 / 显示成 X"时:
   - **先确认连的是哪个库**(dev / 测试库还是生产库),再查——工单给了测试账号就用它定位到那条真实记录。
   - 把**涉及的真实记录、关联表、字段实际值**查出来(如老人→护理员归属查 `caregiver_id` 与当班表 `t_duty_assignment`、用药查 `t_med_batch`),让根因**被真实数据解释**,而不是被"我觉得代码是这么跑的"解释。读代码给出假设 → 连库数据验证假设,两者对上才算坐实。
   - **只读优先**:定位阶段基本都是 `SELECT`,别写库。这一步也常直接坐实"其实是误报 / 缓存 / 环境"(可回到阶段 2 岔路③)。

## 四、阶段 6.2 连库验证(逻辑/数据/口径类必做·详细分步骤)

### 6.2 连库验证(逻辑/数据/口径类必做,强制门禁)

**未连库验真前,不得转「已修复」、不得在留言里声称已修复。** 步骤:

1. **先确认连的是哪个库**:查清当前要连的是 **dev / 测试库**还是**生产库**(别只信 profile——可插独特账号实测,见记忆"先实测后端真实连哪个库")。
2. **重走触发条件,连库看结果**:按工单的复现路径(用其测试账号)重新触发一次,到库里核对——
   - 修的是**值/口径**:查那条记录的字段实际值,是否算成了预期值;
   - 修的是**归属/派单**:查关联表(如 `caregiver_id` / `t_duty_assignment`),是否派给了对的人;
   - 修的是**记录缺失/多写**:查记录数与内容,是否不多不少、副作用可控。
3. **写路径改动尤其验副作用**:是否漏更新、是否多写、是否误伤别的记录/别的老人/别的院区。
4. 验证通过 → 允许进阶段 7 转「已修复」,并把**连库验证的实际结果**写进留言的「回归建议 / 验证结论」里(具体查了哪条数据、值从 X 变成了 Y)。


## 五、阶段 6.3 连库验证的安全红线(防止验证过程本身搞坏数据·带项目案例)


- **只读查询优先**:能 `SELECT` 看清就绝不写库。
- **必须写验证时,用事务内复现 + `ROLLBACK`**:写操作包在事务里,验完立即回滚、**不真正落库**(见记忆里多起"事务内复现 + ROLLBACK",如告警可见性、NEBZ425 CAS 抢单)。
- **严禁在生产库做写验证**:生产库一律只读;确需写验证只在 dev / 测试库进行。
- **测试账号 / 造的数据验证后清理**:临时插的数据、改的测试账号状态,验完清干净,别留脏数据污染后续测试。
