# 云效缺陷工单修复 SOP

## 阶段总览

| 阶段 | 名称 | 一句话 |
|---|---|---|
| 0 | 确保云效 MCP 可用 | 没配就自助配 |
| 0.5 | 模糊请求先搜工单 | 没给工单号才用,列候选让用户选 |
| 1 | 拉取并读懂工单 | 提取四要素、看图、看评论 |
| **2** | **分诊(先置处理中占位 + 早停判定)** | 确定处理即**先置「处理中」占位防重复**,再走三条岔路:①无新说明反复流转→转回创建人 ②产品需求变更→转「已排期」 ③误报/信息不足→转回创建人;都不命中=真 bug→直接往下(状态已在处理中) |
| 3 | 根因定位 | 找到能解释全部现象的根因;逻辑/数据类**连库查真实数据坐实**,不凭感觉 |
| 4 | 对齐目标态 | 预期不明/口径冲突/大重构,先对齐再动手 |
| 5 | 修复 | 最小改动、组件级根治、保留有意行为 |
| 6 | 构建 + 连库验证 | 后台构建验能编译;逻辑/数据类**连库确认结果符合预期**,未验真前不转已修复 ⚠️连库红线 |
| 7 | 工单流转 + 留言 | 转「已修复」+ 结构化留言 ⚠️对外 |
| 8 | 代码提交 | 特性分支、精确 add、**改动按工单分拣(一次 commit 只装一个工单)**、push 需确认 |
| 9 | push 后询问部署 | 主动问一句要不要部署 |
| 10 | 部署到环境 | 手动型流水线要主动触发 ⚠️对外 |

> **主线一句话:**(模糊请求先搜工单)→ 拉取工单 → **确定处理即先置「处理中」占位防重复** → **分诊**(命中任一早停岔路则转走结束)→ 根因定位(**逻辑/数据类连库查真实数据坐实**)→ 对齐目标态 → 修复 → 构建 + **连库验证**(逻辑/数据类**连库确认结果符合预期方可**转已修复)→ 转「已修复」并留言 → 代码提交 → push 后询问是否部署。

## 作业模式:单单亲跑 vs 批量派 subagent(每单一个独立上下文)

> **上面 10 个阶段是"修一个工单"的完整流程。一次要修多个工单时,主 agent 不要在同一个会话里逐单亲跑这 10 个阶段**——上下文会越堆越长、注意力涣散、易胡乱输出(见「贯穿全程的铁律 · 一工单一会话」)。**改用"主会话只调度、每单派一个 subagent"模式**:主 agent 当调度台,把每个工单的整套 SOP 打包塞给一个 subagent 去走,subagent 在**自己的独立上下文**里跑完,只回传简短结论。这样主上下文几乎不增长,修 N 单也无需反复 `/clear`。

**何时用哪种:**
- **单个工单** / 用户只给一个工单号 → 主 agent **亲跑** 10 阶段(默认,不必派)。
- **≥2 个工单批量修** → 用**批量派单模式**:完整 7 条操作规程见 [references/batch-dispatch.md](references/batch-dispatch.md),核心四条——①主会话只调度(判归属 / 写派单 prompt / 汇总结论 / 把需要人定案的决策点收上来),**不亲自**诊断、连库、改码;②派单 prompt **自带全部上下文与红线**(subagent 看不到主线),并要求回传结构化结论;③**多 subagent 改同一仓库每个必用独立 worktree**(`isolation: "worktree"`),互不依赖的工单同一轮并发派;④**决策点(口径冲突 / 大重构 / 对外动作)一律回主会话**,不甩给 subagent 全权拍板。

> **前置依赖:云效(yunxiao)MCP。** 全程靠它读写工单——下文 `get_work_item` / `get_workitem_file` / `list_work_item_comments` / `get_work_item_workflow` / `update_work_item` / `create_work_item_comment` / `get_current_organization_info` 均为云效 MCP 工具,工具名为 `mcp__yunxiao__*`(schema 未加载时用 `ToolSearch` 加载)。**未配置/未连接时,先按下方「阶段 0」自助配置**;organizationId、状态 ID、构建配置等参数也都经它(或当前项目)实时获取,本 skill 不写死(见下「参数获取」)。

## 工具约定

- 需要时用 `ToolSearch` 加载云效 MCP 工具(名形如 `mcp__yunxiao__*`);可用 `Task` / `AskUserQuestion` / `EnterWorktree` 等内置工具。
- 不要把 token 写进仓库——用用户级 MCP 配置或 `.claude/settings.local.json`。

## 贯穿全程的铁律

- **根因要能解释全部现象**,尤其是"为什么偏偏这些条件才触发"——这种条件往往就是定位机制的关键线索,不要满足于一个只能解释部分现象的猜测。
- **⚠️ 逻辑/数据/口径类 bug 必须连库验真,禁止凭感觉修复(强制门禁)**:凡涉及业务口径/算法计算/归属判定/校验规则/状态流转/数据显示不符这类"逻辑问题"(区别于纯前端视觉/交互),**诊断、修复后、门禁三个环节都要连真实数据库确认,只读代码推演不算数**——诊断时连库查真实数据坐实根因,修复后连库确认结果符合预期,**未连库验真前不得转「已修复」/不得声称已修复**;纯前端视觉/交互/样式类可豁免,拿不准从严。**连库时守安全红线**:先确认连的是 dev/测试库还是生产库、只读优先、必须写验证用事务内复现 + `ROLLBACK` 不落库、严禁在生产库做写验证、造的数据验完清理。(三环节完整展开、安全红线的原因、项目案例详见 references/db-verification-and-pitfalls.md,阶段 3/阶段 6 亦有骨架。)
- **开工前先体检工作区、脏就另开 worktree;一次 commit 只装一个工单、改动按工单分拣**(单 agent 也适用):动手前 `git status --short` / `git branch --show-current` / `git worktree list` / `git stash list` 看一眼,不干净就基于干净目标分支(`dev`)另开 worktree 隔离;别让 B 工单改动搭 A 工单的车,混了先分拣、一份都别丢、越界改动单独归它自己工单。(体检/创建/合并/清理/分拣的完整命令与踩坑详见 references/worktree-and-sorting.md。)
- **一工单一会话,别在长上下文里连修多单**:上下文越长,模型注意力越涣散、越容易胡乱输出/答非所问(transformer 固有特性,非模型 bug,1M 窗口能装下 ≠ 装满还保持精度)。所以修完一个工单就 `/clear` 清空、开新会话修下一个,别在同一会话里连续修多个工单硬扛(用户已授权:每单本轮闭环后我主动提醒敲 `/clear`)。**批量修多单时改用"主会话只调度、每单派一个 subagent(独立上下文)"模式**——从根上不让主上下文变长、无需反复清(详见上文「作业模式:单单亲跑 vs 批量派 subagent」)。**进度靠外置载体续,不靠上下文记**:根因/改动/待验证项记进工单留言与项目 memory 文件,新会话从工单 + memory 重新读起,而不是指望模型在长历史里记住前几个工单的细节。单工单内若上下文已很长但没做完,用 `/compact 保留:当前改的文件/已确认根因/待验证项` 压缩而非清空(不丢进度)。大而独立的排查(如全库找某字段所有引用)派给 `Explore`/`general-purpose` subagent,在独立上下文跑、只回传结论,不污染主线。(注:「回复空/答非所问」若是工具调用标签没闭合导致,让主 agent 重发即可,与上下文长度无关。)
- **多页共用的机制,优先组件级根治**,而不是逐页打补丁。
- **现象清楚 ≠ 期望态清楚**:不止视觉/交互类,业务口径/算法/规则类也一样——在"应该改成什么样"不明确时先对齐再动手,别自作主张实现一种方案——返工成本远高于问一句(见阶段 4)。
- **预期与既有设计/已上线逻辑冲突时,先停下来请产品确认,别闷头改**:「预期结果」和已验收上线的逻辑矛盾(算法口径反复变更 / 判重校验规则打架 / 同一工单产品前后口径不一致)是需求决策点而非单纯 bug——把冲突点、必要时候选口径写清楚,留言请产品定案后再实施(详见阶段 4-B)。
- **无新说明的反复流转,直接转回创建人、别硬诊断**:工单又流转到我名下但没追加任何新的打回说明时,别替创建人脑补需求硬诊断,保持「待确认」、转回创建人并留言要信息(详见阶段 2 岔路①)。
- **是产品需求变更而非缺陷的,转「已排期」、别当 bug 修**:诊断后发现「实际结果」就是当前设计的正确行为、「预期」是要改变/新增产品行为(需求而非缺陷),转「已排期」、`assignedTo` 指回创建人/产品并留言,交产品排期(详见阶段 2 岔路②)。
- **认领工单就尽早置「处理中」占位,首要目的是避免多人重复处理**:确定要处理且仍停在未认领态时,诊断前先置「处理中」让团队看到已被认领;分诊后即便命中早停岔路要再转走,来回跳一次可接受(占位收益更大)(详见阶段 2 与收口)。
- **对外/难撤销的动作(工单流转、留言、git push)先确认**再做,除非用户已明确指示。

## 阶段 0 · 确保云效 MCP 可用(缺则自助配置)

本 skill 全程依赖云效 MCP(`alibabacloud-devops-mcp-server`)。**开工前先确认它在**:用 `ToolSearch` 尝试加载 `mcp__yunxiao__get_work_item` 等云效工具;能加载即已配置,直接进阶段 1。

**若加载不到(工具不存在 / 没有 `YUNXIAO_ACCESS_TOKEN`)→ 说明 MCP 没配 → 主动自助配置,别让用户卡住**:按 [references/mcp-setup.md](references/mcp-setup.md) 的步骤操作——引导用户在 <https://devops.aliyun.com> 创建个人访问令牌(权限勾「工作项 / 项目协作」读写),`claude mcp add yunxiao -s user` 完成用户级配置并验证,然后**告知用户重启会话**再从阶段 1 继续(MCP 多在会话启动时加载,别在当前会话硬试)。令牌只存本机安全位置,**勿写进仓库**。

## 参数获取(全部按当前项目 / 组织取值,不写死)

> 本 skill 全局通用,不绑定任何具体项目,也不内置项目专属常量。下列参数一律从当前环境**动态获取**;各组织/项目的工作流与构建配置不同,以实际查询结果为准。

| 项 | 怎么取 |
|---|---|
| organizationId | `get_current_organization_info` 返回的 `lastOrganization` |
| projectId / workItemTypeId | 从 `get_work_item` 返回里现取:`space.id` 作 `projectId`、`workitemType.id` 作 `workItemTypeId` |
| 工单编号 / 链接 | 用户给的工单号(形如 `ABCD-1234`)/ `https://devops.aliyun.com/projex/bug/<工单号>` |
| 缺陷状态 ID | `get_work_item_workflow(organizationId, projectId, workItemTypeId)` 查;本 SOP 会用到多个目标状态:开始处理前置「**处理中**」、修复完成置「**已修复 / Fixed**」、产品需求变更置「**已排期**」、信息不足/待定案保持「**待确认**」——各组织工作流模板的状态名与 ID 不同,一律以查询返回为准(拿不到确切名称时按语义就近匹配,并向用户确认) |
| 构建命令 | 看项目 `package.json` 的 `scripts`(uni-app 微信小程序常见 `build:<env>:mp-weixin` 这类) |
| 构建成功标志 | Vite 项目日志出现 `DONE Build complete.` 且无 error(Dart Sass `legacy-js-api` deprecation 警告是噪声,忽略) |
| 包管理 | 按 lockfile 判断:`package-lock.json`→npm、`yarn.lock`→yarn、`pnpm-lock.yaml`→pnpm |
| 构建产物 | 看项目配置(常见 `dist/`,通常已被 `.gitignore` 忽略,勿提交) |
| 部署流水线 | `list_pipelines(pipelineName=<关键词>)` 找 pipelineId,`get_latest_pipeline_run` / `get_pipeline_run(pipelineRunId)` 查状态,`create_pipeline_run(pipelineId, branch)` 触发。⚠️**别默认「push 会自动部署」**——很多流水线是**手动触发型**(`triggerMode:1` 手动 / `4` API,非 `3` push):最新运行的 commit 若不是你刚推的、或没有 push 触发的新运行,就说明**代码推了但没部署**,dev 跑的还是旧包。**这是 reopen 工单的头号隐藏根因**(见阶段 10 与 references) |

## 阶段 0.5 · 模糊请求时先搜工单(没给具体工单号才用)

用户**直接给了工单号 / 链接** → 跳过本节,直接进阶段 1。
用户**只说「修一下我的 bug」「处理高优先级缺陷」「最近报的那个」**这类模糊请求 → 先用 `search_workitems` 搜出候选,**列给用户选定一个**再进阶段 1,别自己挑。

`search_workitems(organizationId, category="Bug", spaceId=<projectId>)`(三者必填;`spaceId` 即项目 ID,模糊请求下若不知道先 `search_projects` 让用户选)。把用户的自然语言映射成过滤参数:

| 用户说 | 加的参数 |
|---|---|
| 「我的 / 指派给我的」 | `assignedTo: "self"` |
| 「我报的」 | `creator: "self"` |
| 「待确认的」 | `status: "28"`(待确认) 或 `statusStage: "1"`(未开始) |
| 「还没关闭的」 | `statusStage: "1,2"`(排除已关闭) |
| 「高优先级」 | `orderBy: "priority", sort: "desc"` |
| 「最近的」 | `orderBy: "gmtCreate", sort: "desc"` |
| 「标题/描述含 X」 | `subjectDescription: "X"` |

搜索约定:
- **默认 `statusStage: "1,2"`** 排除已关闭项,除非用户要看历史。
- **需要看描述就传 `includeDetails: true`**,一次拿全,省去逐条 `get_work_item`(N+1)。
- `category`/`spaceId`/`status` 的 ID 一律按「参数获取」动态取,**不写死**;`status` 常见值见 `search_workitems` 工具 schema 说明,但仍以 `get_work_item_workflow` 查到的为准。
- 列候选时给「工单号 + 标题 + 优先级 + 状态 + 指派人」,让用户回一个工单号,**选定后再进阶段 1**(对外/修改动作一律等用户定下目标工单)。

## 阶段 1 · 拉取并读懂工单

用云效 MCP 工具读取工单。若 schema 未加载,先用 `ToolSearch` 加载具体工具名(`mcp__yunxiao__get_work_item`、`mcp__yunxiao__list_work_item_comments`、`mcp__yunxiao__get_workitem_file`、`mcp__yunxiao__search_workitems` 等)。不要使用通配名猜测。

1. `get_work_item(organizationId, workItemId="<工单号>")` 取详情。description 是富文本(`htmlValue` / `jsonMLValue`),从中提取四要素:**测试账号、复现步骤、实际结果、预期结果**;有配图 / Figma 链接一并看。**顺手记下 `creator.id`**——阶段 2 转回创建人 / 转已排期都要用。
2. `list_work_item_comments(...)` 看是否有补充信息。
3. **重点读「预期结果」**——它常常直接给出设计意图,一句话锁定修复方向(「预期结果」里一句具体描述往往就能定下方案,省去猜测)。若「预期结果」缺失或含糊,记着到阶段 4 对齐。

**工单内嵌图片**:description 里 `<img>` 的 src 带 `fileIdentifier`(长 hex),不是可直接访问的 URL。流程:
`get_workitem_file(organizationId, workitemId, id=fileIdentifier)` → 返回带签名的临时 OSS URL(约 1 小时过期) → `curl -sL "<url>" -o /tmp/x.png` → 用当前环境的图片查看能力查看。URL 过期就重取。并排对比图可用 PIL 裁剪左右半再放大看。

读懂工单后,**先进阶段 2 分诊,别直接跳去写代码**。

## 阶段 2 · 分诊(早停判定 + 置处理中)

> **先占位、再判定。** 拉取工单后,只要确定要处理它(用户已点名/选定)、且它当前仍是「待处理 / 待确认」等未认领态,**先把状态置「处理中」占位——首要目的是避免多人重复处理**(分诊本身要读代码/看图/复现,有耗时窗口,不占位期间别的同事可能同时认领同一单)。占位后再进下面的分诊判定。
>
> **在根因定位(阶段 3)之前,先判定这单到底要不要开发修。** 云效工单被 reopen / 反复打回是常态,还常有"伪装成 bug 的需求"和误报——不先分诊就一头扎进诊断,极易做无用功、改错方向。三条早停岔路**依次判定**,命中任一就转走结束本轮(走阶段 7 里对应的转出动作,状态从「处理中」再转到对应目标态——来回跳一次可接受,占位防重复的收益更大);三条都不命中,确认真 bug,直接进阶段 3(状态已在「处理中」)。

### 岔路① · 无新说明的反复流转 → 转回创建人

判定条件(**同时满足**才算命中):
- **当前 `assignedTo` 又是"我"**(工单再次流转到当前处理人名下);
- **本轮流转没有伴随新评论说明**——即创建者 / 测试在把工单打回给我之后,**没有追加任何新评论**讲清"这轮为什么又打回、具体要改什么、上轮修复哪里没达标"。判据:`list_work_item_comments` 里最后一条评论,不是本轮 reopen 的新说明(要么评论区为空,要么最后一条评论早于/无关于最近这次流转,只是历史留痕)。
  - **精确核对(时间戳没有现成 API 对齐时用)**:`list_workitem_activities(organizationId, workItemId)` 拉活动记录,找到**最近一次 `assignedTo` 变更成"我"**的那条活动及其时间;再拿 `list_work_item_comments` 里最后一条评论的时间比对——**评论时间早于该次流转时间(或压根没有评论)→ 判定"本轮无新说明",命中**;评论时间晚于该次流转、且内容是本轮打回说明 → 不命中,继续往下判。内容语义靠人工读一眼(时间只是初筛,别只看时间戳就下结论)。

命中处理:保持状态**「待确认」**、`assignedTo` 转回创建人、留言要信息(具体见阶段 7 岔路①转出)。

> **反面教训**:没有新说明还硬着头皮重新诊断,等于替创建人脑补需求——极易改错方向、再被 reopen,来回空转。缺信息就要信息,别猜。

### 岔路② · 产品需求变更(非缺陷) → 转「已排期」

> 与阶段 4-B(预期与既有逻辑「冲突」需产品拍板哪个对)不同:本节是口径不冲突,只是**当前行为本就是设计的正确行为,工单的「预期」是要求改变/新增产品行为**——一个伪装成 bug 的需求。
> ⚠️ 这条判定**常需先读一点代码确认现状口径**才能下结论(否则分不清"跑偏了要修"还是"本就如此想要更多")。所以实操上可以:先花少量精力定位相关逻辑确认现状,若坐实"现状即正确设计",就在此命中、转已排期,**不必**走完整根因定位/修复。

判定命中(满足其一,且已读代码确认现状即正确设计):
- 「实际结果」描述的现象,经定位就是**当前设计的预期行为**(代码按既定规则跑,没有偏离设计的缺陷);
- 「预期结果」要的是**新增能力 / 改变现有规则 / 加字段加流程**,而非把跑偏的行为纠回设计;
- 正文更像"希望产品支持 XX""建议改成 XX""增加 XX 功能",而非"XX 坏了 / 报错 / 与设计不符"。
- 典型例:NEBZ-400《入库药品问题》——描述的入库行为符合现有设计,诉求是调整产品规则,属需求变更。

命中处理:状态转「**已排期**」、`assignedTo` 指回创建人/产品、留言说明(具体见阶段 7 岔路②转出)。

> 拿不准是「阶段 4-B 口径冲突留待确认」还是「本节需求变更转已排期」时:核心区分是**现状是否偏离设计**——偏离设计要修的是 bug、或口径要拍板(阶段 4);现状即正确设计、只是想要更多/不同,是需求(本节转已排期)。实在难分就把判断依据讲清、`AskUserQuestion` 让用户定。

### 岔路③ · 误报 / 无法复现 / 信息不足 → 转回创建人

- **误报**:现象其实是操作路径、缓存、环境等非缺陷因素造成(参见记忆里多起"缓存/操作路径误报"案例);
- **无法复现 / 信息不足**:按工单给的账号、步骤复现不出,或关键信息缺失,无法定位。

命中处理:保持状态**「待确认」**、`assignedTo` 转回创建人、留言点明结论并要补充(具体见阶段 7 岔路③转出)。

### 收口 · 三条都不命中 → 确认是真 bug → 直接进阶段 3

- 状态**已在分诊前置为「处理中」占位**(见本阶段引言),这里无需重复置,直接进阶段 3 做根因定位与修复即可。
- **置「处理中」的操作细节**(引言里占位、或本阶段收口若尚未置时执行):`get_work_item_workflow(...)` 查到「处理中」状态 ID → `update_work_item(organizationId, workItemId, updateWorkItemFields={ "status": "<处理中状态 ID>" })`。状态名/ID 各组织不同,按「参数获取」动态取。
- 让团队/报告人从工单上就能看到它已被认领、正在处理,而不是停在「待确认/待处理」看不出进展,别的同事也不会重复认领。
- 属对外操作,但**用户已点名/选定要处理该工单时,置「处理中」视为已获许可、直接执行**(占位防重复优先,不必再逐次确认);仅当尚未明确要处理该单时才先确认。
- **占位后若分诊命中早停岔路**(需转回创建人 / 转已排期),照阶段 7 对应动作把状态从「处理中」再转到目标态即可——来回跳一次是可接受的代价。

## 阶段 3 · 根因定位

1. 按复现路径定位涉及的**页面**与其依赖的**公共组件 / mixin / store**;逻辑/数据类还要定位后端的 **Service / Mapper / SQL / 字段**。
2. 形成根因假设,标准是**能解释工单里的每一个现象**。把"触发条件"当线索:像"必须先上滑、且经过进二级页再返回才出现"这种组合条件,往往直指某个具体生命周期或状态回灌逻辑。
3. **⚠️ 逻辑/数据/口径类:连库查真实数据坐实根因,别只靠读代码推演**(强制,见「贯穿全程的铁律」)。工单说"值不对 / 派给了错的人 / 少了记录 / 显示成 X"时:先确认连的是 dev/测试库还是生产库(用工单测试账号),把涉及的真实记录、关联表、字段实际值查出来,让根因**被真实数据解释**(读代码给出假设 → 连库数据验证假设,两者对上才算坐实);定位阶段只读优先(基本都是 `SELECT`),这一步也常直接坐实"其实是误报/缓存/环境"(可回阶段 2 岔路③)。详细步骤与项目案例(归属表、业务主键字段、某批次表等)详见 references/db-verification-and-pitfalls.md。
4. 判断范围:是单页问题,还是多页共用机制的系统性问题。若是后者,记下"还有哪些页面会受同样影响",阶段 5 一并根治。

## 阶段 4 · 对齐目标态(视觉/交互类 + 业务口径/规则类,都看)

「实际结果」说的是"现在错成什么样",但修复需要知道"应该是什么样"。当「预期结果」不够具体(没指明颜色 / 参照页 / 交互方式),**或预期本身可能与既有设计/已上线逻辑冲突**时,**先对齐,再写代码**:

**A. 视觉/交互类**(没指明颜色 / 参照页 / 交互方式):
- 找**参照页**:项目里有没有同类、已经做对的页面?拿它当对照基准。
- 找**设计稿**:视觉类优先看 Figma / 设计稿;工单配图常常就是"设计 vs 实际"对比。
- 拿不准就**给候选方案并向用户确认**(可用 `AskUserQuestion`)。别自己挑一种实现。

**B. 业务口径/算法/规则类**(预期与既有设计或已上线逻辑冲突,或产品口径前后反复):
- **先读代码确认现状口径**:工单说"该值不对"时,先定位计算/校验逻辑,搞清现在线上到底按什么口径/规则算,以及涉及哪个字段——别凭工单一句话就改。
- **把冲突摆上台面**:若新预期与已验收上线的逻辑矛盾(如"算法口径又改了""判重规则与既定行为打架"),或本次改动会改变某用户已上线字段的显示结果,**把矛盾点连同数据写清楚,留言请产品定案**;必要时列候选口径让产品逐项确认,并要求"定稿、不再反复"。
- **确认后再动**,且把"历史数据是否回刷""未维护/无数据如何兜底显示"等边界一并问清——这些常是二次返工的来源。
- ⚠️ **口径冲突/需产品定案时,别只留言——还要把负责人指回创建人(见阶段 7 第 5 点)**,否则工单仍挂开发名下、报告人不会主动来定案,球没真正踢回去。

> 反例一(视觉):工单报告某处"显示穿帮",没对齐预期就自选了一种实现,结果用户要的是另一种方案——白白返工一轮。
> 反例二(口径):工单"目标差距值不对",产品在工单里几轮改口径、甚至推翻自己上一轮已验收的版本;若每次都按最新一句话直接改,会反复返工、且可能让某用户已上线的显示值被悄悄改掉。正确做法是先读代码定位现状口径,把"新口径会推翻上一版、会改变某用户显示结果"讲清楚,留言请产品定案后再实施。

**C. 工单其实是「大重构 / PRD」而非单点 bug**:有的"缺陷"工单正文是一整份需求文档(含背景/目标/范围/多个待确认项),或诊断后发现要**跨多端 + 后端、删页 / 改数据源 / 破坏性变更**。这类**不能当 bug 一把梭**:①先产出**方案文档**(边界决策清单 + 跨端接口契约 + 三端改动清单 + 分阶段实施顺序),把「待产品拍板的决策点」逐条列清、留言请产品定案;②确认后**分阶段实现**——后端接口先行(前端依赖它)、前端各端并行、**下线/删除类放最后且走 Gate**(读路径全切完 + 相关数据清零 + 分两步:先摘入口观察、再禁接口,保留代码可回滚);③老逻辑下线前先 grep 全仓残留引用与写入来源,甄别哪些接口仍被新主入口活跃调用(不能盲删)。这种工单一轮做不完,逐阶段 commit + 汇报。

## git worktree 隔离修复(摘要,细节详见 references/worktree-and-sorting.md)

- **开工前先体检主工作区**(单 agent 也做):`git status --short` / `git branch --show-current` / `git worktree list` / `git stash list` 看一眼;脏就基于干净的目标分支(`dev`)另开 worktree,别把自己的改动和别人的在途工作缠进同一个 stash/分支。
- **多 subagent 改同一仓库必用独立 worktree**(`Agent`/`Task` 工具传 `isolation: "worktree"`),不同仓库天然隔离无需 worktree;**一次 commit 只装一个工单,越界改动分拣归位、一份都别丢**。
- 创建/软链 node_modules/合回目标分支(ff vs 非 ff)/清理判定/按工单分拣的完整命令与踩坑教训,全部见 references/worktree-and-sorting.md。

## 阶段 5 · 修复

- 改动**最小且零副作用**。
- 动公共组件前,先 `grep` 出**谁在用**,评估影响面(一个公共组件被十几个页面引用很常见)。
- 优先**修正错误的默认行为**,而非到处加特例;翻转默认值这类大影响面改动,要逐一确认每个受影响页面都是正向变化。
- **保留有意为之的既有行为**:把"少数页面特意要的效果"显式声明出来(对确实需要该效果的个别页面单独开启),不要被一刀切误伤。

## 阶段 6 · 构建 + 连库验证 ⚠️ 含连库红线

> **构建通过 ≠ 修好了。** 构建只证明"能编译",逻辑/数据类 bug 是否真修好,必须**连真实数据库确认结果符合预期**——这是把工单转「已修复」的**强制门禁**(见「贯穿全程的铁律」)。

### 6.1 构建(验能编译)

- 后台跑测试构建命令(用 `run_in_background` 启动并轮询),不要用前台 `sleep` 假等待。**首次全量构建较慢(可能数分钟),改几个文件的增量构建通常几十秒**。
- 成功标志:`DONE Build complete.`、无 error。失败就如实贴日志,别声称已修好。
- 后端(Maven)则跑编译打包命令,以能编译通过为准。

### 6.2 连库验证(逻辑/数据/口径类必做,强制门禁)

- **未连库验真前,不得转「已修复」、不得在留言里声称已修复**——按工单复现路径(用其测试账号)重走一次,到库里核对结果符合预期(值/口径、归属/派单、记录缺失或多写),写路径尤其验副作用。
- 分步骤(先确认连哪个库 / 重走触发看结果 / 验副作用 / 把实际结果写进留言)与项目案例(归属字段、归属表、某批次表等)详见 references/db-verification-and-pitfalls.md。

### 6.3 连库验证的安全红线(防止验证过程本身搞坏数据)

- 只读查询优先;必须写验证时用事务内复现 + `ROLLBACK` 不落库;严禁在生产库做写验证(生产库一律只读);测试账号/造的数据验完清理。
- 每条红线的原因与项目案例(事务内 ROLLBACK、CAS 抢单验证等)详见 references/db-verification-and-pitfalls.md。

### 6.4 UI 走查(前端类)

- 纯前端视觉/交互/样式类可豁免连库。UI 类缺陷最终仍需在微信开发者工具按复现路径走查——把这一步作为给用户的回归建议写进工单。

## 阶段 7 · 工单流转 + 留言 ⚠️ 对外操作

> 团队可见的动作,除非用户已明确要求(如"工单改一下"),否则先确认再做。
> 本 SOP 涉及的工单状态流转有四类目标态,按前面判定的结论选一:**处理中**(阶段 2 收口,已在动手前置好)、**已修复**(修完,下面 1-4)、**已排期**(产品需求变更,岔路②转出)、**待确认**(信息不足/待定案,岔路①③转出 + 第 5 点)。状态 ID 一律 `get_work_item_workflow` 动态查。

### 修好了 · 转「已修复」

> **⚠️ 转「已修复」的前置门禁**:逻辑/数据/口径类 bug **必须已完成阶段 6.2 连库验证、且结果符合预期**,才能转此状态。只跑通构建、没连库验真的,不算修好,不得转。

1. `get_work_item_workflow(organizationId, projectId=space.id, workItemTypeId=workitemType.id)` 查可流转状态及其 ID。
2. `update_work_item(organizationId, workItemId, updateWorkItemFields={ "status": "<已修复状态 ID>" })` 流转为**已修复**(状态 ID 取自上一步查询结果)。
3. `create_work_item_comment(...)` 留言,**用纯文本、不要带 emoji**(云效留言含 emoji 会报 400),**结构化**写清便于测试验证:
   - **根因**:一句话讲透机制(逻辑/数据类点明是靠**连库查真实数据**坐实的)
   - **修复方案**:改了什么、为什么这么改
   - **影响范围**:哪些页面/场景被修复,哪些行为刻意保留
   - **验证结论**(逻辑/数据类必写):连库验证的**实际结果**——查了哪条数据、值/归属从 X 变成了 Y,证明已修好(而非仅"构建通过")
   - **回归建议**:具体到可操作的复现路径
4. 若修复范围在过程中扩大(从单页扩到组件级),**补一条评论同步最新范围**,避免之前的留言误导测试。

### 阶段 2 命中早停岔路的转出动作

> 共同底线:**只留言不转负责人 = 工单仍挂开发名下,报告人不会主动看到它待自己处理,球没真正踢回去,工单僵死。** 所以早停岔路一律「转状态 + 转 `assignedTo` + 留言」三件事一起做。`creator.id` 取自阶段 1。属对外操作,先向用户确认再转(除非用户已明确「照 SOP 自动处理」)。

5. **岔路①③(无新说明反复流转 / 误报 / 信息不足)→ 转回创建人,保持「待确认」**:
   - `update_work_item(organizationId, workItemId, updateWorkItemFields={ "assignedTo": "<creator 的 userId>" })`。
   - **状态别乱转**:属"等报告人补信息/确认"的,**保持「待确认」**最贴切(不要转「已修复」——没修;也别急着转「暂不修复」,那偏"我方判定不修",除非报告人已认可)。
   - 留言(纯文本、无 emoji)按岔路点明:①"本轮工单再次流转到我名下,但未见新的复现说明/打回原因,无法判断需要修改什么。已转回负责人 XXX,请补充本轮的复现步骤、实际 vs 预期、或上轮修复未达标之处;补齐后再转回开发。" / ③写清"经复现/排查,该现象为 XX(误报/操作路径/无法复现/信息不足),非代码缺陷。已转回负责人 XXX,请 YY(确认/补充账号步骤);如仍需修改再转回开发。"
6. **岔路②(产品需求变更)→ 转「已排期」+ 指回创建人/产品**。与第 5 点「待确认」区别:第 5 点是信息不足/待报告人处理;本条是已确认它是需求、要交给产品排期实现。
   - `update_work_item(organizationId, workItemId, updateWorkItemFields={ "status": "<已排期状态 ID>", "assignedTo": "<creator 的 userId>" })`——「已排期」状态 ID 按「参数获取」查(无同名态则就近匹配语义等价态并向用户确认)。
   - `create_work_item_comment(...)` 留言(纯文本、无 emoji)点明:此工单为产品需求变更而非缺陷,当前行为符合现有设计,诉求需作为需求排期实现;已流转「已排期」并转回负责人 XXX,请纳入需求排期,排期确定后如需开发再转回。例:NEBZ-400《入库药品问题》。

## 阶段 8 · 代码提交(push 需显式确认)

1. 确认分支——**按「撞车风险」决定,不是每单都建特性分支**:
   - **默认:单工单 + 单 agent + 工作区干净且就在 `dev` → 直接在当前 `dev` 分支改并 commit**(无需另建 feature 分支;这也是团队日常小改的常态,后端 push `dev` 即触发流水线部署)。
   - **仅下列「会撞车」的情况才隔离到独立分支/worktree**:①多个 subagent 并行改同一仓库;②开工前工作区已脏(有别人在途改动 / 停在别人的 feature 分支);③大重构/PRD 需多阶段多次 commit。判据见 references/worktree-and-sorting.md。
   - **任何情况都不直接提交到 `main`**。
   - ⚠️ feature 分支是「隔离脚手架」不是「每单必留物」:成果最终落 `dev`,合并/清理后即删,别攒一堆用完不删的冗余分支。
2. `git status` 核对;**精确 `git add <具体文件>`**(`dist` 虽已忽略,仍逐个指定,避免误带无关改动)。
3. commit message 遵循项目惯例:`type(scope): 简述 <工单号>`
   - type 如 `fix` / `style` / `chore`;scope 取改动所在模块(如 `components`)
   - 一个 commit 可带多个工单号
   - **例**:`fix(components): 关闭某公共组件的错误默认行为,修复二级页显示异常 ABCD-1234`
4. 提交后**用 `git log` 核实 commit 确实产生了**——别想当然(曾遇到命令回显像是"被拒绝"、实则已执行的情况,核实一眼即可)。
5. **不擅自 `git push`**。提交后告知推送命令(新分支首推 `git push -u origin <branch>`),由用户决定。

## 阶段 9 · push 完成后主动询问是否部署

用户确认 push 后(不擅自推,但推送动作完成即视为本阶段触发点),**主动**用 `AskUserQuestion` 问一句:"是否需要触发部署流水线到测试/开发环境?"——不要默认执行,也不要默认跳过。
- 用户选**是** → 进入阶段 10 部署。
- 用户选**否** → 结束本轮,照旧告知"需要部署时请找我或自己触发",不再重复追问。

这一步只是「主动提个醒」,不改变阶段 10"默认不部署"的底线——部署仍需用户在此处明确点头,不是每次 push 都自动触发流水线。

## 阶段 10 · 部署到环境(经阶段 9 确认,或用户直接要求「部署 / 上线」时) ⚠️ 对外操作

> 部署是有一定影响面的操作,仍需用户明确同意(经阶段 9 确认,或用户直接说「部署一下」)才执行,**不因为 push 成功就自动触发**。确认要部署后,**先核实是否真的需要触发**,别想当然。

1. **别默认 push 自动部署**。云效流水线很多是**手动触发型**——push 后不会自动跑。核实方法:`list_pipelines(pipelineName=<仓库名关键词>)` 找 pipelineId → `get_latest_pipeline_run` 看最新运行的 `triggerMode`(`1` 手动 / `4` API / `3` 才是 push 触发)和它部署的 commit。**若最新运行不是你刚推的 commit、也没有 push 触发的新运行 → 代码推了但没部署,环境跑的是旧包**。
2. **需要部署且是手动型** → 经用户确认后用 `create_pipeline_run(organizationId, pipelineId, branch="dev", comment="...")` 触发,拿到返回的 `pipelineRunId`。
3. **轮询到部署完成**:`get_pipeline_run(pipelineRunId)` 直到 `status:SUCCESS`(构建 stage + 部署 stage 都 SUCCESS,部署 stage 的 deployOrder `status:Success` 才算重启完)。别用前台 `sleep` 干等,用回查/后台轮询。`FAIL` 则取构建日志(结果里有 `LogPipelineJobRun` 的 downloadUrl)报告用户。
4. **多端部署要「后端先行」**:前端详情页/家属端依赖后端新接口,后端没部署好前先推前端会有短暂不一致窗口(拿旧数据/报错)。顺序:先部署后端 → 确认 SUCCESS → 再部署前端(前端各端之间可并行)。
5. **这正是 reopen 工单头号根因**:上轮"已修复"评论在,代码也在远程 dev,但流水线手动型没人点部署 → 环境旧包 → 测试复现如初 → reopen。复查工单先按此排查(见阶段 7 与 references)。
